# Bioinformatics Agent 项目说明书

## 一、项目背景与问题定义

### 1. 应用场景

本项目面向生物医学领域的科研人员、研究生和临床医生，帮助用户高效检索、解析和对比 PubMed 学术文献。

### 2. 当前痛点

- **全文阅读成本高**：生物医学文献多为长篇幅英文论文，逐篇精读耗时巨大。
- **关键信息难定位**：图表数据、实验结论、靶点信息等分散在长文中，人工提取效率低。
- **跨文献对比效率低**：多篇文献的方法、结果、结论需要人工整理对比，容易遗漏细节。
- **检索门槛高**：PubMed 高级检索语法复杂，非图情专业人员难以构建精准检索式。

### 3. 项目目标

开发一个基于 AI Agent 的生物医学文献智能分析系统，实现：

- 用户输入自然语言研究问题，Agent 自动规划检索策略并执行
- 自动检索 PubMed 文献，筛选最相关的论文
- 对每篇文献提取结构化信息（目标、方法、靶点、关键发现等 17 个字段）
- 跨文献对比分析，发现共性、矛盾和研究空白
- 生成结构清晰、来源可溯（标注 PMID）的文献综述报告
- 实时展示 Agent 执行过程，增强用户对 AI 输出的信任度

### 4. 边界说明

当前版本不解决的问题：

- 不支持本地 PDF 上传解析（仅使用 PubMed 摘要）
- 不支持多轮对话式交互（每次查询独立执行）
- 不包含用户注册/登录功能（已预留接口）
- 不支持多语言输出（报告仅中文）

## 二、Demo 功能范围

### 1. 输入形式

用户在聊天界面输入自然语言研究问题，例如：

- "对比近三年针对 EGFR 靶点的 NSCLC 临床研究中 PD-1 抑制剂的疗效与安全性"
- "CRISPR-Cas9 基因编辑技术在遗传性疾病治疗中的最新临床进展"
- "PD-1/PD-L1 抑制剂联合化疗对比单药治疗在晚期肺癌中的疗效 Meta 分析"

同时可设置"最大文献数"参数（2-20 篇）。

### 2. 输出形式

- **思考过程**：实时展示 Agent 的执行步骤（检索、筛选、解析、对比等），类似大模型的"思维链"
- **最终报告**：Markdown 格式的文献综述，包含研究背景、文献逐篇介绍、对比分析、结论与建议
- **来源可溯标签**：报告中每篇文献标注 `[PMID:xxxxx]`，点击可查看文献详情
- **文献详情面板**：右侧滑出面板，展示论文标题、作者、摘要、结构化提取数据

### 3. 核心功能

| 功能模块 | 说明 |
|---------|------|
| 任务规划 (Planner) | LLM 将用户意图拆解为 4-7 个子任务的 DAG（检索→解析→对比→报告） |
| 文献检索 (Search Worker) | LLM 翻译中文描述为 PubMed 检索式，pymed 检索，LLM 筛选排序 |
| 文献解析 (Parser Worker) | 对每篇文献提取 17 个结构化字段（目标、方法、靶点、关键发现等） |
| 对比分析 (Compare Worker) | 跨文献对比，找出共性、矛盾、研究空白 |
| 报告生成 (Reporter) | 汇总所有分析结果，生成结构化 Markdown 报告 |
| 对话管理 | 支持新建对话、切换对话、对话历史持久化（localStorage） |
| 流式输出 | 通过 SSE 实时推送执行状态，前端动态更新 |

### 4. 功能边界

- 当前仅使用 PubMed 摘要，未接入全文 PDF 解析
- 不支持用户上传本地文件
- 不支持流式报告逐字生成（报告为一次性生成）
- 检索失败时自动重试（宽泛关键词），但不支持用户手动调整检索策略

## 三、系统设计

### 1. 系统架构

```
┌─────────────────────────────────────────────────────────┐
│                    Frontend (React)                      │
│  Vite + TypeScript + Tailwind CSS + lucide-react        │
│  localhost:5173                                         │
└────────────────────────┬────────────────────────────────┘
                         │ HTTP + SSE
                         ▼
┌─────────────────────────────────────────────────────────┐
│                   Backend (FastAPI)                      │
│  uvicorn + sse-starlette                                │
│  localhost:7860                                         │
└────────────────────────┬────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│                  Agent Engine                            │
│                                                          │
│  ┌──────────┐    ┌──────────────────────────────────┐   │
│  │ Planner  │───>│  SubTask DAG (4-7 tasks)         │   │
│  └──────────┘    └──────────────────────────────────┘   │
│                         │                                │
│          ┌──────────────┼──────────────┐                │
│          ▼              ▼              ▼                │
│  ┌──────────────┐ ┌──────────┐ ┌────────────┐          │
│  │Search Worker │ │Parser W. │ │Compare W.  │          │
│  └──────┬───────┘ └────┬─────┘ └─────┬──────┘          │
│         │              │              │                  │
│         ▼              ▼              ▼                  │
│  ┌─────────────────────────────────────────────┐        │
│  │            Tools Layer                       │        │
│  │  PubMed API (pymed) │ DeepSeek LLM │ PDF    │        │
│  └─────────────────────────────────────────────┘        │
└─────────────────────────────────────────────────────────┘
```

### 2. 模块说明

| 模块 | 文件 | 输入 | 处理逻辑 | 输出 |
|------|------|------|----------|------|
| **Planner** | `backend/agent/planner.py` | 用户查询 | DeepSeek LLM 拆解为子任务 DAG | SubTask 列表 (4-7个) |
| **Search Worker** | `backend/agent/workers/search_worker.py` | 子任务描述 + 用户意图 | LLM 构建 PubMed 检索式 → pymed 检索 → LLM 筛选 | 筛选后的文献列表 |
| **Parser Worker** | `backend/agent/workers/parser_worker.py` | 筛选后的文献 | LLM 从摘要提取 17 个结构化字段 | 结构化数据 (LiteratureSchema) |
| **Compare Worker** | `backend/agent/workers/compare_worker.py` | 多篇文献的结构化数据 | LLM 跨文献对比分析 | 对比报告 (共性/矛盾/空白) |
| **Executor** | `backend/agent/executor.py` | 用户查询 + 参数 | 调度 Planner → Workers → Reporter，流式 yield 状态快照 | SSE 事件流 |
| **API Server** | `backend/api/server.py` | HTTP 请求 | FastAPI 路由 + SSE 流式推送 | JSON 快照流 |
| **useResearch** | `frontend/src/hooks/useResearch.ts` | 查询 + 参数 | 连接 SSE 端点，管理流式状态 | React 状态 |
| **useChat** | `frontend/src/hooks/useChat.ts` | 消息操作 | 对话 CRUD + localStorage 持久化 | 对话列表 |

### 3. 执行流程

```
用户输入查询
    │
    ▼
[Planner] LLM 拆解为 4-7 个子任务 DAG
    │
    ▼ (yield 快照 #1)
[Search Worker]
    ├── LLM 翻译中文 → PubMed 检索式
    ├── pymed 调用 PubMed API
    ├── LLM 筛选最相关文献
    │
    ▼ (yield 快照 #2)
[Parser Worker]
    ├── 遍历每篇文献
    ├── LLM 提取 17 个结构化字段
    │
    ▼ (yield 快照 #3)
[Compare Worker]
    ├── 构建结构化数据摘要
    ├── LLM 跨文献对比
    │
    ▼ (yield 快照 #4)
[Reporter] LLM 汇总生成最终 Markdown 报告
    │
    ▼ (yield 最终快照 + [DONE])
前端渲染完整报告
```

### 4. 评估指标

| 维度 | 评估方式 | 说明 |
|------|----------|------|
| 任务完成情况 | 定性 | 输入 3 个典型研究问题，检查是否完整走完检索→解析→对比→报告流程 |
| 工具调用正确性 | 定性 | 检查 execution_log 中 PubMed 检索式是否合理、筛选结果是否相关 |
| 结果质量 | 定性 | 报告是否结构清晰、引用准确、对比分析有深度 |
| 运行稳定性 | 定性 | 执行过程中是否出现 SSE 断连、LLM 超时、解析错误 |
| 过程可检查性 | 定性 | 思考过程（思考链）是否实时展示每一步操作 |
| 检索召回率 | 定量 | 对已知答案的查询，检查检索到的文献是否覆盖关键论文 |

## 四、当前开发中出现的问题与处理

### 1. 模型输出不稳定

- **问题**：DeepSeek LLM 返回的 JSON 偶尔格式不规范，导致解析失败。
- **处理**：使用 `response_format: "json_object"` 强制模型返回 JSON；`chat_json()` 方法内置 fallback，解析失败时返回 `{"raw": raw_text}` 而非抛异常。

### 2. PubMed 检索中文查询失败

- **问题**：用户输入中文研究问题，pymed 无法直接检索中文关键词。
- **处理**：Search Worker 增加 LLM 翻译步骤，将中文描述转为标准 PubMed 英文检索式（含 MeSH 词和字段限定符），并增加宽泛关键词重试机制。

### 3. PDF 全文解析未接入

- **问题**：`parser_worker.py` 中 `pdf_available` 字典始终为空，仅使用摘要。
- **处理**：当前阶段优先保证端到端流程跑通，PDF 解析列为后续计划。MinerU + PyMuPDF 双引擎解析器已实现（`tools/pdf_parser.py`），待接入 PMC 全文下载。

### 4. 前端依赖体积大

- **问题**：初始构建 JS bundle 778KB，含 reactflow 等重型依赖。
- **处理**：重构为聊天界面后移除 reactflow 引用，bundle 降至 639KB。

### 5. 环境依赖问题

- **问题**：conda 环境中 `urllib3` 版本过旧，`urllib3.packages.six.moves` 模块缺失。
- **处理**：升级 `urllib3` 到 2.x 版本解决。
